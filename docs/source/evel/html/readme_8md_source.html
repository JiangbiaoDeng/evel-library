3 # Introduction
4 
5 The ECOMP Vendor Event Listener ("EVEL") library encapsulates the use of
6 AT&T's JSON API to the collector function within the ECOMP infrastructure.
7 
8 As such, it provides a reference implementation of the EVEL JSON API which
9 can either be directly as part of a project or can be used to inform the
10 independent implementation of an equivalent binding to the API in another
11 development environment.
12 
13 This section provides an overview of the library and how it is integrated
14 into the target application. If all you want is a set of instructions to
15 get you started, the @ref quickstart "Quick Start" section is for you. If
16 you want a more in-depth understanding of the _EVEL Library_ then this section
17 provides an overview and then you can read the detailed API documentation for
18 each function. The documentation for evel.h is a good starting point, since
19 that defines the public API of the _EVEL Library_.
20 
21 # Library Structure
22 
23 The API is designed to be used on multi-process platforms where each process
24 may be multi-threaded. Each process using this library will create an
25 independent HTTP client (using libcURL). Each process will have a single
26 thread running the HTTP client but that thread receives work on a
27 ring-buffer from however may threads are required to implement the function.
28 
29 **Note**: libcurl imposes a constraint that it is initialized before
30 the process starts multi-threaded operation.
31 
32 # Typical Usage
33 
34 The library is designed to be very straightforward to use and lightweight to
35 integrate into projects. The only serious external dependency is on libcURL.
36 
37 The supplied Makefile produces a single library **libevel.so** or
38 **libevel.a** which your application needs to be linked against.
39 
40 Each process within the application which wants to generate events needs to
41 call ::evel_initialize at the start of day (observing the above warning
42 about not being MT safe at this stage.) The initialization specifies the
43 details of where the API is located. Management of configuration is the
44 responsibility of the client.
45 
46 Once initialized, and now MT-safe, there are factory functions to produce
47 new Faults (::evel_new_fault) and new Measurements (::evel_new_measurement).
48 
49 The Fault and Measurement structures are initialized with mandatory fields
50 at the point of creation and optional fields may be added thereafter. Once
51 set, values in the structures are immutable.
52 
53 Once the event is prepared, it may be posted, using ::evel_post_event, at
54 which point the calling thread relinquishes all responsibility for the
55 event. It will be freed once successfully or unsuccessfully posted to the
56 API. If, for any reason, you change your mind and don't want to post a
57 created event, it must be destroyed with ::evel_free_event.
58 
59 Finally, at the end of day, the library can be terminated cleanly by calling
60 ::evel_terminate.

62 ## Example Code
63 
64 The following fragment illustrates the above usage:
65 
66 ```C
67 
68  if (evel_initialize(api_fqdn,
69  api_port,
70  api_path,
71  api_topic,
72  api_secure,
73  "Alice",
74  "This isn't very secure!",
75  EVEL_SOURCE_VIRTUAL_MACHINE,
76  "EVEL demo client",
77  verbose_mode))
78  {
79  fprintf(stderr, "Failed to initialize the EVEL library!!!");
80  exit(-1);
81  }
82 
83  ...
84 
85  fault = evel_new_fault("My alarm condition",
86  "It broke very badly",
87  EVEL_PRIORITY_NORMAL,
88  EVEL_SEVERITY_MAJOR);
89  if (fault != NULL)
90  {
91  evel_fault_type_set(fault, "Bad things happen...");
92  evel_fault_interface_set(fault, "My Interface Card");
93  evel_fault_addl_info_add(fault, "name1", "value1");
94  evel_fault_addl_info_add(fault, "name2", "value2");
95  evel_rc = evel_post_event((EVENT_HEADER *)fault);
96  if (evel_rc != EVEL_SUCCESS)
97  {
98  EVEL_ERROR("Post failed %d (%s)", evel_rc, evel_error_string());
99  }
100  }
101 
102 ```
103 
104 The public API to the library is defined in evel.h. The internal APIs
105 within library are defined in separate headers (<em>e.g.</em>
106 evel_internal.h), but these should not need to be included by the code
107 using the library.

182 # Restrictions and Limitations
183 
184 ## Constraint Validation
185 
186 The _EVEL Library_ has been designed to be production-safe code with the
187 emphasis at this stage being in correctness of operation rather than
188 raw performance.
189 
190 The API tries to check as much information as possible to avoid misuse and
191 will **assert()** if constraints are not satisfied. This is likely to lead
192 to the rapid discovery of coding errors by programmers, but does mean that
193 the application can fail abruptly if the library is misused in any way.
194 
195 ## Performance
196 
197 The default Makefile avoids aggressive optimizations so that any core-files
198 are easy to interpret. Production code should use greater optimization
199 levels.
200 
201 As described above, the HTTP client is single threaded and will run all
202 transactions synchronously. As transactions are serialized, a client that
203 generates a lot of events will be paced by the round-trip time.
204 
205 It would be a straightforward enhancement to use the multi-thread API into
206 libcurl and use a pool of client threads to run transactions in parallel if
207 this ever became a bottleneck.
208 
209 ## Logging
210 
211 The initialization of the library includes the log verbosity. The verbose
212 operation makes the library very chatty so syslog may get rather clogged
213 with detailed diagnostics. It is possible to configure syslog to put these
214 events into a separate file. A trivial syslog.conf file would be:
215 
216 ```
217 
218 # Log all user messages so debug information is captured.
219 
220 user.* /var/log/debug
221 ```
222 
223 If verbose logging is enabled, the cURL library will generate information
224 about the HTTP operations on **stdout**.
